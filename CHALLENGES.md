# Challenges & Setup Notes

This document records the notable setup steps and problems encountered while
building the Turuq backend, and how they were resolved. It doubles as the
"challenges faced" write-up requested in the assessment.

---

## 1. Creating the MongoDB database (Atlas)

We used **MongoDB Atlas** — MongoDB's managed cloud database — on the free **M0**
tier. This avoids a local database install and lets the *same* database be reused
in production when the API is deployed.

**Steps taken:**

1. Created an Atlas account + organization at [cloud.mongodb.com](https://cloud.mongodb.com).
2. Inside a **Project**, deployed a free **M0 cluster** (`Cluster0`).
   - Gotcha: **Clusters, Database Access, and Network Access live at the _Project_
     level, not the _Organization_ level.** If you only see billing/users/settings
     in the sidebar, you're at the Org level — open **All Projects → your project**
     to get the Project sidebar.
3. **Security → Database Access** → created a database user (`turuq_app`) with an
   autogenerated password.
4. **Security → Network Access** → allowed access from anywhere (`0.0.0.0/0`) so
   both local development and the future deployment host can connect. The database
   user credentials still protect the data.
5. **Database → Connect → Drivers → Node.js** → copied the connection string into
   `.env` as `MONGODB_URI`, adding the database name `turuq` to the URI.

### MongoDB vs MySQL note (for a relational background)

Unlike MySQL, you do **not** run `CREATE DATABASE` / `CREATE TABLE`. MongoDB
creates the database (`turuq`) and its collections **lazily** — they first appear
only after the app writes its first document. So right after connecting, the
`turuq` database may not be visible in a GUI yet; that is expected, not a bug.

Rough vocabulary map:

| MySQL / Laravel      | MongoDB / Mongoose        |
| -------------------- | ------------------------- |
| Database             | Database                  |
| Table                | Collection                |
| Row                  | Document (JSON-like)      |
| Column               | Field                     |
| Eloquent model       | Mongoose model            |
| `CREATE TABLE` (DDL) | (implicit — schema in app) |

## 2. `mongodb+srv://` fails on this network — `queryTxt ETIMEOUT`

### Symptom

Connecting with the Atlas-provided string
`mongodb+srv://...@cluster0.geiem8o.mongodb.net/turuq` failed immediately with:

```
queryTxt ETIMEOUT cluster0.geiem8o.mongodb.net
```

### Background: what SRV and TXT records are

DNS is not only "name → IP address" (that is an **A record**). A single DNS name
can hold several *kinds* of records:

- **A record** — name → IPv4 address (the usual one).
- **SRV record** ("service" record) — name → a *service location*: which host(s)
  and port(s) provide a given service. It exists so clients don't have to
  hardcode server addresses.
- **TXT record** — arbitrary text attached to a name; commonly used for domain
  verification and to carry small pieces of configuration.

The `mongodb+srv://` scheme is a convenience wrapper. Instead of hardcoding the
cluster's three replica-set servers and their options, the driver asks DNS at
connect time:

- an **SRV lookup** to discover the three server hostnames + port `27017`, and
- a **TXT lookup** to fetch default connection options (`authSource=admin`,
  `replicaSet=atlas-...`).

### Diagnosis

- `nslookup -type=SRV ...` returned the three servers on **both** the default DNS
  and Google DNS (`8.8.8.8`) → **SRV works**.
- The **TXT** lookup timed out, and forcing Node to use `8.8.8.8` / `1.1.1.1` via
  `dns.setServers()` did **not** help.
- Conclusion: something in the network path (ISP — **Orange Egypt** home Wi‑Fi
  router) silently drops DNS **TXT** queries. Because `mongodb+srv://` cannot
  proceed without the TXT answer, the connection fails.

### Fix: use the standard (non-SRV) connection string

We replaced the SRV URI with the explicit **seed-list** form, which needs no SRV
or TXT lookup — only ordinary A‑record resolution (which works fine):

```
mongodb://turuq_app:<password>@ac-goizrtz-shard-00-00.geiem8o.mongodb.net:27017,ac-goizrtz-shard-00-01.geiem8o.mongodb.net:27017,ac-goizrtz-shard-00-02.geiem8o.mongodb.net:27017/turuq?ssl=true&replicaSet=atlas-5cplch-shard-0&authSource=admin&retryWrites=true&w=majority
```

- The three hosts came from the successful SRV lookup.
- `replicaSet` and `authSource` are the values normally stored in the TXT record;
  the replica-set name was obtained from a successful connection's `hello.setName`.

After this change the server connects successfully:

```
Server listening on http://localhost:5000 [development]
```

### Trade-offs

- The standard string works on **any** network, so it is safe to keep.
- Its only downside: the shard hostnames are written out explicitly. If Atlas ever
  changed them (rare for a stable cluster) the string would need updating.
  `mongodb+srv://` auto-adapts, and will likely work fine on a deployment host
  (Render/Heroku) whose network does not block TXT.
